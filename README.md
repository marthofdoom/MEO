# Marth's Enchanting Overhaul (MEO)

A full replacement for Skyrim SE enchanting that turns enchantments into
**gems**: pulled from found gear, socketed into your own equipment anywhere
with no station, and **leveled through use**. Weapons, armor, and jewelry gain sockets; staves, wands, and
scripted artifacts are detected and left completely alone.

Built entirely from Linux with no Creation Kit and no xEdit: the plugin is
generated byte-by-byte from Python, the gem catalog is parsed straight out of
the game masters, Papyrus compiles under Proton wine, and the planned native
layer builds on GitHub Actions.

> **Status: early development (P0 prototype).** The architecture is being
> validated in-game one load-bearing assumption at a time before the full
> system is built on top of it. See **Roadmap** below and `Docs/DESIGN.md`.

## The idea

- **Enchantments become gems.** Every generic vanilla enchantment collapses
  into a gem *family* (the catalog is generated from ~740 ENCH records in the
  five masters, deduplicated by effect). Weapon gems are on-hit effects; armor
  gems are constant effects.
- **Socket anywhere, lose nothing.** A Gem Pouch power opens the socket menu
  with no crafting station. Socketing and unsocketing your own gems is always
  free, lossless, and reversible.
- **Gems level I -> V through your kills**, then a mastered gem *births* a
  fresh copy of itself — the only way to replicate one. Soul gems get a
  permanent second life as XP fuel.
- **Sockets scale with the enchanting tree.** Chest is dual-socket by default;
  the penultimate perk adds a second socket to gloves and boots, the final one
  to weapons and jewelry. Linked sockets enable **support gems** (All, Added
  Effect, Elemental, ...) that modify the normal gem they share an item with.
- **The item is the database.** Socket state lives in each item's own runtime
  enchantment (a hidden marker effect encodes gem + level), so there is no
  side-table to desync and, in the native design, no Papyrus save bloat.

## Why native (planned)

The endgame is a small CommonLibSSE-NG SKSE plugin alongside the ESP + Papyrus
shell, for two reasons: **conformance** (Address-Library-based, engine-version
resilient — the strict modern standard) and **no save bloat** (the socket/XP
index serializes to the SKSE co-save, not the Papyrus VM). The toolchain and
hook doctrine are inherited from the sibling MRO project, not re-derived.

## Repo layout

```
MEO_GenerateESP.py   generates MEO.esp (never hand-edited)
data/                gem catalog parsed from the game masters (JSON)
tools/               parse_ench.py, compile.sh, release.sh
Source/Scripts/      MEO Papyrus sources
Source/Stubs/        compile-only stubs for base types absent from SKSE sources
Docs/                START AT Docs/INDEX.md
releases/vX.Y.Z/     immutable release archive (never deleted)
```

## Documentation

**`Docs/INDEX.md` is the entry point.** The docs are written so any capable
person or model can continue development from them alone: the design spec, a
binary-format guide (every record recipe verified against vanilla bytes), a
symptom -> fix debugging cookbook, and the in-game test matrices.

## Roadmap

- **P0 — prototype (current, `v0.0.1-p0`).** Minimal plugin: a Fire gem I-V
  socket/level/remove loop that validates runtime-enchantment persistence
  (the "item is the database" premise) across save/load, re-equip, and rebuild.
- **P1 — core.** Full generated gem catalog, loot conversion, XP / leveling /
  birthing, soul feeding, the enchanting-perk rework, and an MCM.
- **P2 — support gems.** Echo / Conduit / Focus and dual-socket linking.
- **P3 — packaging.** FOMOD, plus a Requiem / LoreRim compatibility plugin.
- **P4 — uniques.** Curated gems from artifact enchantments.

## Requirements

SKSE64, SkyUI, powerofthree's Papyrus Extender. Core targets vanilla Skyrim SE
+ SKSE; a Requiem / LoreRim compatibility plugin is planned. Developed against
the LoreRim toolchain on Skyrim SE 1.6.
