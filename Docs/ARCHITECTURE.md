# Architecture — the native subsystem map (v1.0.6)

How `native/plugin.cpp` (~6060 lines, one translation unit) is actually
organized: per subsystem, what it does, the load-bearing functions with line
refs, and the engine flow each one rides. **DESIGN.md** says what MEO is;
**ENGINE_NOTES.md** says what the engine does; this file says where OUR code
does it. Line refs are against v1.0.6 (branch `debug/vendor-sound-probe`) —
they drift, the function names don't. The rules this code must obey live in
**INVARIANTS.md**; the mistakes it must not repeat in **ANTI_PATTERNS.md**.

## 0. Identity & data structures (plugin.cpp:99-260)

The whole mod hangs off ONE map:

```
g_sockets : unordered_map<InstKey, SocketRecord>        (:119)
SocketRecord { string gid; uint8 level; float xp; }     (:102)
InstKey = (baseFormID << 24) | (uniqueID << 8) | slot   (MakeKey :116)
```

- **Bit layout**: slot = bits 0–7, uid = bits 8–23, base FormID = bits 24–55.
  `kMaxSockets = 2` (:110); loose-gem records always slot 0. SaveCallback
  decomposes the key with the same shifts (:5756).
- **`gid`** is the stable catalog string ("fire", "chaos", …) — never an
  index (ENGINE_NOTES §6: indexes don't survive catalog changes).
- **`MintUID`** (:128) mints uids in MEO's reserved range **[0x9000, 0xFFFF]**
  (`g_nextUID` :120), re-floors on uint16 wrap, and scans all `kMaxSockets`
  slots for collision before returning — a wrapped uid can never clobber a
  live record via `operator[]` (review find S3). Engine-assigned uids below
  0x9000 are kept as-is when the instance already has one (StampInstance
  :1543 — the key includes baseID, so they can't collide across bases).
- **Catalog**: `g_gems` (vector of `ResolvedGem` :170/:180) is
  `native/GemCatalog.h` (generated by `tools/gen_catalog_header.py`) resolved
  against the live load order in `ResolveCatalog` (:819) at kDataLoaded, then
  overlaid by the installer's `meo_calibration.json` via `LoadCalibration`
  (:724): per-gid riders (`g_calibration`), per-level source enchants
  (`g_calLevels`), per-list magnitude curves (`g_calCurves`), and the
  conversion table (`g_calConversions` → `g_convert`). Lookups: `g_gemByItem`
  (MISC FormID → {gemIdx, level}), `g_gemByGid` (:190-191).
  **Phase 3 minted families**: the calibration's additive `"minted"` section
  is parsed PER ENTRY (one bad family costs only itself — never the brittle
  all-or-nothing families block) into `g_calMinted`, then a second
  `ResolveCatalog` pass registers each as a full runtime family:
  `GemDef` owned by `g_mintedDefs` (a `std::deque` — addresses stable, so
  `ResolvedGem::def` and `g_gemByGid` string_view keys can alias it), items
  = the ESP's reserved pool MISCs (`kPoolFloor` 0xB00 guard: a corrupt
  calibration can never capture curated forms; forms already bound to
  another family are rejected loudly), names forced at runtime from the
  live MGEF (the baked "Uncut Gem" is only the no-calibration face).
  `ResolvedGem.minted` = conversion-only: every spawn/vendor/NPC pool loop
  and `ConduitSibling` skip it. Missing MGEF (stale calibration after mod
  removal) **or any unresolved pool form (stale ESP)** registers DISABLED —
  the curated missing-master posture; co-save records keyed by the minted
  gid go dormant, never destroyed. `MenuUnsocket` and the swap-evict check
  the level's item form BEFORE erasing a record (INVARIANTS 27) so a
  dormant family's gem is refused, not silently destroyed.

## 1. The enchant-build pipeline

Every socket, unsocket, level-up, and load funnels through one function.

- **`RebuildInstanceEnchant(base, xList, owner=nullptr)`** (:1275) — reads
  ALL slots of `(base, uid)` from `g_sockets`, builds ONE created enchantment
  via the engine's own `BGSCreatedObjectManager::AddWeaponEnchantment` /
  `AddArmorEnchantment` (:1459 — the SKSE recipe, ENGINE_NOTES §3), forces
  the display name through `ExtraTextDisplayData` + the engine's
  `GetDisplayName(base, health)` reconcile (:1510-1523, rename traps
  ENGINE_NOTES §2). No filled normal slot → strips enchant + name extras
  (:1342). Steps inside:
  1. **Slot collection** (:1304): supports (Echo/Conduit/Focus) are collected
     apart — a support is only live when LINKED (exactly 1 support + 1 normal,
     :1329). Off-domain normal gems are erased unless a Conduit adapts them
     (:1339 — a weapon MGEF riding armor could damage the wearer).
  2. **Stacking cap** (m35c): `applyCap = worn && (!owner ||
     owner->IsPlayerRef())` (:1293). `WornActiveEffectKeys()` (:1132) scans
     the PLAYER's worn gear and returns the top-2-by-level instance keys per
     gid; a capped 3rd+ copy contributes no effect (:1324). **The owner gate
     exists because the scan is player-only**: an NPC-worn key is never in
     the set, so an ungated cap strips every NPC enchant (build-B1 — see
     INVARIANTS). The whole applyCap/owner mechanism is scheduled for
     retirement by the runtime tally-cap (ROADMAP-1.0.7-tally-cap.md).
  3. **Magnitude** (:1421): `GemBaseMag(def, lvIdx)` (:1095 — installer
     per-list curve `g_calCurves` if present, else compiled catalog default)
     × `g_magnitudeMult` (MCM master) × `(1 + 0.05 × g_attuneRank)` ×
     `GemPerkMult` (:1103 — affinity/Facet +25% each, Chaos matches all three
     elements) × Focus boost or Conduit transfer ratio. Riders (m21/m22
     recipe companions) ride at `absMag`-scaled or `primaryMag × ratio`
     (:1438); Echo weapon-AoE sets `effectItem.area` (:1427).
  4. **Effect array**: `BSTArray<RE::Effect>` sized up-front (:1399-1409),
     one primary Effect per filled socket + riders; every `eff.cost = 0`.
  5. **m38c pricing** (:1465-1492): the engine prices enchanted items as
     `base + fEnchantmentPointsMult × MaxCharge` (weapons) or `base +
     fEnchantmentEffectPointsMult × EnchantmentCost` (armor). Premium =
     `Σ kSocketTierValue[tier]` (:114, {40,75,120,190,300}) ×
     `g_socketValueMult`. Weapons keep `costOverride = 0` (must never drain)
     and carry the premium in **charge**; constant-effect armor never spends
     charge, so it carries the premium in **costOverride** (kCostOverride
     flag always set — it's part of MEO's effect signature, see §8). The old
     flat `charge=0xFFFF` is what ballooned weapons to 20k gold.
- **`StampInstance(base, xList, gemIdx, level, slot, xp, owner)`** (:1535) —
  writes the `g_sockets` record (mint uid if absent), then calls
  RebuildInstanceEnchant **forwarding the owner** (:1559). Every worn-stamp
  caller must pass the real owner (INVARIANTS) — `MaybeStampNPCGear` passing
  nullptr was build-B1: the NPC's fresh enchant was stripped and its record
  orphaned in the co-save permanently.
- **Activation**: building the extra is not enough on worn gear. In-session:
  `ApplyWornAbility` (:1655, `Actor::UpdateWeaponAbility`/`UpdateArmorAbility`)
  when the extra CHANGED; the full idempotent teardown is `EquipCycleWorn`
  (:1681, guarded by `g_equipCyclingBases` :1679) — ENGINE_NOTES §8 has the
  whole m14→m35d saga. `SocketCapacity` (:1070): gloves 2, chest 2 with
  Twinned, weapons 2 with Jeweler, else 1.

## 2. Socket bookkeeping, the pouch, transfers

- **The Gem Pouch** is a hidden CONT ref (`kPouchContID` 0x8FE :363), created
  once per save (`EnsurePouchRef` :383, ref id persisted co-save v7),
  `PouchRef()` (:376) resolves it. `RouteGemsToPouch` (:516) moves loose
  catalog gems from player inventory into it; `RecoverStrandedGems` (:414)
  reclaims gems from a lost pouch ref (gated by `g_pouchCreatedThisLoad`).
  `GiveGemInstance` (:2388) returns an unsocketed gem as a real instance via
  the engine mint (PlaceObjectAtMe → stamp → PickUpObject, with
  `SetOwner(player)` first — theft trap, ENGINE_NOTES §9).
- **Container transfers rewrite uids** (ENGINE_NOTES §1). The
  `ContainerSink` (:5156) fires `RekeyTransferredSockets` (:4558): match the
  arriving orphan xList with the stranded record and move it to the new uid;
  the event's uniqueID field names the ARRIVING uid. Ambiguous multi-instance
  transfers log + skip.
- **Menu operations** (all deferred to SKSE tasks via `QueueMenuTask` :3075):
  `MenuSocket` (:2770), `MenuUnsocket` (:2698), `DestroyGem` (:2734),
  `FeedSoulToGem` (:2986, `kSoulFeedXP` :621, Soul Feeder doubles).
  `FindInstanceXList` (:2363) is the canonical re-find — never cache an
  `ExtraDataList*` across frames.

## 3. Kill sink & the XP economy

Flow: `DeathSink` (:4842, act on `dead==true`, killer = player or teammate)
→ boss/dragon detect (LocRefType "Boss" / ActorTypeDragon → `g_bossXPMult`)
→ **defers to a task by handle** → `AwardKillXP` (:2252) → `GrantGemXP`
(:2183) per filled worn slot → possible level-up → mastered birth.

- **`AwardKillXP(owner, ap)`** (:2252): multipliers — Mentor Gem in inventory
  ×2 (:2258), ×(1 + Enchanting AV/100) (:2269, the "Enchanting grows gems"
  spine, DESIGN §6), Gem Cutter ×1.5 player-only (:2272). Then the
  **snapshot pattern** (build-S3/xp-S1, :2275): collect every (worn item,
  filled slot) into a `targets` vector FIRST, then award — because
  `GrantGemXP` can synchronously `AddObjectToContainer` (mastered birth) and
  run an equip cycle, both of which mutate the `entryList` being walked. Each
  award **re-finds the g_sockets record by key** (:2330) — a prior target's
  level-up may have rewritten the map.
- **`GrantGemXP`** (:2183): guards `level < 1` (corrupt data would underflow
  `kXPThresholds[level-1]`, xp/hooks-S4); supports cap at III and only earn
  while linked (:2188-2196). On level-up: rebuild (owner-forwarded), then
  worn-activation — the cheap `ApplyWornAbility` unless the gid has >2 worn
  copies (`WornGidCount` :1177) where the cap must redistribute via
  `ReapplyWornSockets(true,true)` (:2213). Level 5 births a level-I copy of
  the gem into the owner's inventory (:2238).
- **Enchanting SKILL XP model (m37/v1.0.6)** — every rate is a **baked base
  constant × a 1.0-default MCM multiplier**, master-gated by `fEnchSkillXP`
  (`GrantEnchantingXP` :2109 — `mult <= 0` kills all of it):

  | Source | Baked base | MCM slider (×) | Where |
  |---|---|---|---|
  | Kill trickle | `kGemKillSkillXP = 0.001`/GemXP pt (:1756) | `fGemKillXpMult` (INI-renamed! see §9) | AwardKillXP :2341, once per kill |
  | New family discovered | `kDiscoverSkillXP = 10` (:1755, was 50) | `fDiscoverSkillXP` | DiscoverGem :2126 |
  | Soul fed | `kSoulSkillXP[tier] = {11,26,56,94,150}` (:3057, v1.0.6 −25%) | `fEnchSkillXP` (direct, gated :3058) | FeedSoulToGem |
  | Gem destroyed | `g_destroySkillXP = 20 × level` (dev INI) | — | DestroyGem :2765 |
  | Gem leveled | `g_levelSkillXP = 12 × new level` (dev INI) | — | GrantGemXP :2224 |

  (Distinct from GEM XP per fed soul: `kSoulFeedXP[5] = {1, 2.5, 5, 12, 40}`
  :621, ×2 with Soul Feeder. `FeedSoulToGem` is the one Enchanting-XP call
  that bypasses `GrantEnchantingXP` — it still gates and multiplies by
  `g_enchSkillXPMult` itself.)

- **Discovery** (co-save v11): `DiscoverGem`/`CheckGemDiscoveries` (:2120,
  :2133) — first possession of a family = one-time XP + notify, persisted in
  `g_discoveredGems`. Pre-v11 saves seed silently (`SeedDiscoveries` :2155,
  `g_needSeedDiscoveries`) so upgrades don't fire an XP burst.

## 4. Loot, spawns, conversion (the gem economy)

- **Corpse gems**: `RollCorpseGem` (:4023) — player kills only, RNG
  `g_gemDropChance`, level II at `g_gemLevel2Chance`. Support gems:
  `RollBossSupportGem` (:4045) — boss/dragon kill, player level ≥
  `g_supportMinLevel`, tier I only.
- **World weapons born socketed**: `MaybeStampWorldWeapon` (:4062) from
  `CellAttachSink` (:4776, fires PER REFERENCE). Rolls are **deterministic**:
  `HashU32(refID ^ 'MEO1')` (:4083) — a given ref decides once, forever,
  nothing stored. Skips already-enchanted/artifact/bound bases
  (`IsSocketableWeaponBase` :1602 — base `formEnchanting` must be null; a
  socketed item's ONLY enchant source is the instance extra).
- **NPC worn gear**: `MaybeStampNPCGear` (:4463) — humanoids with base
  Aggression ≥ 1 (enemies only), deterministic per-ref roll, themed pool by
  `DetectArchetype` (:4100, race keyword + dominant base skills →
  `g_npcPool[arch][isArmor]` weighted by `kArchThemeW` :205). The gem is
  LIVE on the enemy (stamped with the NPC as owner — build-B1); at death it
  drops loose.
- **Conversion** (m23 — covered enchant families CONVERT to socketed):
  `ConvertInstanceEnchant` (:4173, effect-signature match `SameEffectSig`
  :4162), `ConvertInventory` (:4267 — actors AND containers; container path
  uses PlaceObjectAtMe → stamp → `AddObjectToContainer(fromRefr=tempRef)`,
  the m37 vendor fix), `ConvertWorldRef` (:4412). Sweeps: player at
  post-load; arrivals via `ContainerSink`; vendor chest at DialogueMenu-open
  (`StockVendorGems` :4675 — once per vendor per game day, max 3 gems, only
  when no gem is already in stock; `g_vendorStockedDay`) **plus** the
  two-frame-deferred BarterMenu-open sweep in `MenuSink` (:2005-2070) —
  the engine re-rolls LVLI stock at barter-open with NO container event
  (ENGINE_NOTES §15 in the cross-repo copy; §9 vendor entries here), then the
  open list is rebuilt with the engine's own inventory-update reloc
  (51911/52849; NG 3.7.0 predates the header helper) aimed at the menu's
  bound ref + vendor + chest.
- **Hand-placed support gems**: `TryPlaceHandPlaced` (:4757), bitmask
  co-save v10.

## 5. Perks (FormID block 0x810–0x81C in MEO.esp)

Constants at :628-636: Attunement 1–5 = 0x810–0x814, Gem Cutter 0x815, Soul
Feeder 0x816, Twinned 0x817, Jeweler 0x818, Pyre/Frost/Stormstone
0x819–0x81B, Facet Insight 0x81C. `RefreshPerks` (:1843):

- **Two modes**: `g_treeMode` (installer's `MEO - Patch.esp` present →
  perks come from the rewritten enchanting tree, cost perk points) vs
  auto-grant by Enchanting skill (0/20/40/60/80 for Attunement; gate levels
  mirror the tree reqs, :1856-1867).
- Effects are read from **cached booleans/rank** (`g_attuneRank`,
  `g_hasGemCutter`, …) refreshed on JournalMenu close + load — consumers are
  `RebuildInstanceEnchant` (magnitude), `AwardKillXP` (Cutter),
  `FeedSoulToGem` (Feeder), `SocketCapacity` (Twinned/Jeweler).
  `bDebugAllPerks` (:1888) forces the CACHE only — no perks are added, so
  switching it off reverts cleanly.
- `ApplyTemperPerk` (:5294): `bTemperNoPerk` grants/revokes Arcane Blacksmith
  (revocable — `g_meoGrantedArcane` co-save v8 records that WE granted it).

## 6. Hooks, threads, locks

MEO installs exactly **three trampoline call-hooks**, all in
`menuhook::Install` (:3963, `SKSE::AllocTrampoline(256)`, called from
`SKSEPluginLoad` :6037 before the renderer exists), each a `write_call<5>`
(:3960) — details/IDs in ENGINE_NOTES §9:

| Hook | RelocationID | Thread it puts us on |
|---|---|---|
| D3DInit | (75595, 77226)+0x275/0x9 | init (once) |
| DXGIPresent | (75461, 77246)+0x9 | **render** — draws ImGui, `TickEnchHumMute` |
| InputDispatch | (67315, 68617)+0x7B | input — translates/swallows while menu open |

`menuhook::Install()` is the FIRST thing `SKSEPluginLoad` does (:6040) —
call-sites must be written before the renderer initializes; the 1.6.1170
version check after it is warn-only. The present thunk runs
`TickEnchHumMute()` BEFORE the `g_d3dReady`/menu-open gate, so the sound
restore survives an ImGui init failure. Everything else is event sinks
(registered at kDataLoaded, `OnMessage` :5942) — no vanilla-function
detours.

**Thread/lock map** (the part reviews keep probing):

- Engine mutation happens ONLY on the main thread via
  `SKSE::GetTaskInterface()->AddTask` (sinks and menu actions defer;
  `QueueMenuTask` :3075).
- `g_menu` snapshot mutex: the render thread draws a snapshot rebuilt by
  main-thread tasks (`BuildMenuSnapshot` :2437).
- **`g_imguiIoMx`** (:3140, xp/hooks-S2): ImGui's IO event queue is not
  thread-safe — input thread pushes `Add*Event`, window thread calls
  `ClearInputKeys` (WndProc), render thread drains in `NewFrame`. One mutex
  serializes every ImGui-IO touch across all three; held only around the IO
  block, never across the engine passthrough (no cross-lock with the
  snapshot mutex).
- `g_magMx` + atomics: the sound-mute window (§7).
- Detached `std::thread` sleepers (`RunDeferredReapply` :5699, the 15s
  fallback :5714) do nothing but sleep and re-queue onto a task.

## 7. Windowed enchant-hum mute (:262-360)

The approved hand-write exception #1 (marth): the stacked
MAGEnchantedUnsheathe hum on every runtime build is silenced by maxing the 4
SNDR forms' `staticAttenuation` (`kEnchHumSndr` :272, `kSilentAttenuation`
0xFFFF) during a **deadline window**, then restoring the saved originals
exactly. Protocol:

- Any MEO action (rebuild :1458, reapply :5559, vendor convert) calls
  `OpenEnchHumMuteWindow(ms)` (:339): CAS-max the atomic deadline
  `g_soundMuteUntilMs`, then `ApplyEnchHumMute` (:296) immediately
  (idempotent under `g_magMx`; saves originals once).
- Load path: kPreLoadGame arms a 30s blanket BEFORE deserialization;
  kPostLoadGame *shortens* via `SetEnchHumMuteDeadline(4000)` (:350) —
  deadline set BEFORE apply so a racing present tick can't see
  muted+expired and restore early (:5979).
- **Single restorer**: only the render-thread `TickEnchHumMute` (:355)
  restores, and `RestoreEnchHumMute` (:317) re-checks the deadline UNDER the
  lock (a concurrent re-open must win). Outside the window the vanilla cue
  is untouched. (The v1.0.6+ call-site hook experiments to replace this live
  on this branch's probe commits — see memory/roadmap, not yet landed.)

## 8. Load, co-save, reactivation

**Serialization** (`kSerID 'MEO1'`, record `'GEMS'`, **schema v11** :151-153):

- `SaveCallback` (:5728) writes: nextUID, starter/mentor/armorStarter flags,
  pouch ref id (v7), arcane flag (v8), scaffold (v9), handPlacedMask (v10),
  discovered-gid set (v11, length-prefixed strings), then count + per-record
  `{baseID, uid, slot(v5), level, xp, gid}`.
- `LoadCallback` (:5772) — the four disciplines (INVARIANTS §co-save):
  1. **ResolveFormID on every stored FormID**: socket bases (:5904 — a
     changed load order remaps plugin indices; unresolved → drop the record,
     its plugin left) AND the pouch ref (:5834 — dynamic FF refs still pass
     through the co-save handle map).
  2. **Short-read bail** (N2): if the socket count itself can't be read, stop
     — never fabricate keys from garbage (:5867).
  3. **Clamp deserialized values**: level → [1,5] at the source (:5891) so
     `kXPThresholds[level-1]` can't underflow anywhere downstream.
  4. **Newer-version records**: SKSE does NOT round-trip unread co-save data
     — saving with an older DLL DESTROYS them. Log + one-time in-game warning
     box; the only protection is not saving (:5794-5812).
  `RevertCallback` (:5926) zeroes everything save-scoped.
- **Reactivation** (worn instance enchants are delivery-dead after load until
  a real equip — ENGINE_NOTES §8): `ScheduleReapplyWornSockets` (:5708) runs
  an immediate rebuild+diag task, then arms a **generation token**
  (`g_reapplyGen`/`g_reapplyPending` :5696, review S2 — a plain bool let a
  first load's 15s fallback steal a second load's reapply). Consumers:
  LoadingMenu-close in `MenuSink` (:1991, `exchange(0)` claims it) schedules
  equip-cycle passes at +5s/+12s; the 15s fallback CAS-consumes only its own
  generation (:5719). `ReapplyWornSockets` (:5549) collects targets first,
  then `EquipCycleWorn` each (m35d: the idempotent engine teardown — retired
  blinkless reapply could accumulate abilities), then backstop dispel +
  followers.
- **`DispelStaleGemEffects`** (:5358) — the orphan/dedup sweep, player-only,
  **allowance model** (m38e/S4): allowance per (FF-enchant-form, MGEF) = how
  many active effects the WORN items carrying that enchant can legitimately
  produce; only a worn xList — or one whose base is mid-equip-cycle
  (`g_equipCyclingBases`, closes the m26e race) — vouches. Our effects are
  fingerprinted as FF-form + `kCostOverride` flag + gem-MGEF membership
  (:5446-5453; NOT costOverride==0 — armor carries value there since m38c).
  Copies beyond allowance → `ae->Dispel(true)` (engine call, M2-clean). The
  1.0.7 tally-cap re-points this same machinery (see ROADMAP).
- **`ReapplyFollowerSockets`** (:5485, m36n): same collect-then-cycle shape
  over `ProcessLists::highActorHandles` teammates — targets held by
  ActorHandle and re-resolved at cycle time (review B2).
- **Echo follower-share** (:4907, heartbeat :5025): approved hand-write
  exception #2 — ONE persistent ESP spell (`kEchoShareSpellID` 0x809) whose
  single effect is rewritten each 8s tick (:4978-4985) and re-cast on
  followers via `CastSpellImmediate`, dispel-before-recast so it's always
  exactly 1×. Self-expiring (12s > heartbeat): follower saves only ever
  reference a real form; removing MEO just lets it lapse.

## 9. Configuration (:1733-1836)

- Two files, last-wins: `Data/SKSE/Plugins/MEO.ini` (dev/seed) then
  `Data/MCM/Settings/MEO.ini` (MCM Helper's persisted store) —
  `ReadConfig` (:1823), re-read live on JournalMenu close and gem-menu open.
  `ApplyIniFile` (:1766) strips MCM Helper's UTF-8 BOM and **skips
  unparseable values** rather than applying 0.0 (N2).
- All keys + defaults are the globals at :1733-1760. `bEnableLogging` flips
  the spdlog level in one place (:1829).
- **The rename rule** (INVARIANTS): MCM Helper persists stored VALUES per
  key name forever. `fGemXpSkillXP` was an absolute rate (0.01); v1.0.6 made
  it a ×multiplier (default 1.0) and therefore **renamed it
  `fGemKillXpMult`** (:1813) — a stale persisted 0.01 read as a multiplier
  would silently zero the kill trickle. Any future raw→multiplier semantic
  change must rename, and keep the MCM config `.py`/json key in lockstep.

## 10. Generators, installer, release (the off-DLL half)

- **ESP generator** `MEO_GenerateESP.py` (hand-rolled binary TES4 writer —
  MANUAL_MOD_CREATION_GUIDE.md is the format reference). Emits `out/MEO.esp`,
  `meo_runtime.json`, the MCM Helper `config.json`, and `MCM/Settings/MEO.ini`
  (utf-8-sig — hence the BOM strip in `ApplyIniFile`). **The FormID band
  0x800–0x8FF is a DLL contract**: 0x800/0x801 marker MGEFs, 0x802/0x803
  pouch MGEF/SPEL, 0x804 startup QUST, 0x805–0x807 FLSTs, 0x808 MCM QUST,
  0x809 Echo share SPEL, 0x810–0x81C the perk block, 0x8FE pouch CONT,
  0x8FF Mentor MISC; gem MISCs from 0x900, allocation anchored by
  `data/gem_forms.frozen.json` — regen only unions new gid×level pairs,
  `next_fid = max+1`, never recycles. Support gems belong to NEITHER FLST
  (the DLL keys off the catalog's `isSupport`). Phase 3 reserved pool: 160
  placeholder MISC "Uncut Gem" forms at **0xB00–0xB9F** (32 slots × 5 levels,
  EDID `MEO_PoolNN_L`, neutral amethyst mesh, value 0, in NO FLST —
  conversion-only per marth 2026-07-16), anchored by
  `data/pool_forms.frozen.json` (second frozen anchor, kept separate from
  `gem_forms.frozen.json` so `allocate_gems`' max-scan never sees pool fids).
  The installer's slot-assignment step (Phase 3 later stages) reads this
  anchor and appends assignments per user machine; curated catalog growth
  caps at 0xAFF (~47 more families of headroom).
- **Catalog header** `tools/gen_catalog_header.py` → `native/GemCatalog.h`
  (`kGems[]`, 58 entries; `GemRider[2]` fixed arrays; `kXPThresholds`
  {500,1000,3000,8000}). Hard-exits on a gid missing from its THEME map or
  >2 riders — those guards protect the DLL's fixed-size arrays.
- **Installer / Synthesis patcher** (`installer/`, C# Mutagen;
  `Commands.cs` is SHARED between the standalone exe and the Synthesis
